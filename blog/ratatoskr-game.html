Project Brief

13 March 2025

My idea is a story based maze and text game. The player is the squirrel Ratatosk from Norse mythology, carrying messages up and down the world tree. It’s fed up, or bored, or something, and wants to destroy the world by making the entities (eagle and dragon) whose messages it carries really angry.

I chose to do this because I like the idea of a tree climbing explorer game where you chat to characters, and this was a good excuse to work on it. Also an excuse to research viking insult poetry and do pixel art and write a story on engineering time. Challenges are likely to be collision avoidance with the wall and figuring out the best way to tie cutscenes to locations.

The plan is to have a maze which you have to navigate towards certain locations, such as Nidhogg’s lair and the eagle’s nest, potentially starting at Mimir's well at the base of the tree, to give introduction. There will be cutscenes in those locations where you talk with the characters, and maybe some choice based conversation affecting the game. If I have time and space left, maybe Thor is there to throw some lightning at you too.

For the purposes of the project, the map shouldn’t be too big, makes it easier to test and the game takes less time. Also only 1 short cutscene for each entity. So I need to make the story as concise as possible.

Randomness:

For the minigame, I need randomness, so I’ve started to research how to achieve it. I used the linear congruential method (Linear Congruence method for generating Pseudo Random Numbers | GeeksforGeeks) and built in Mbed timer, timing the duration of intro cutscene (involves button input so good for randomness) and using that as the seed for my randomness generator. The method for generating randomness is shown below: 


t.start();
cutscene.cutscene_intro();
t.stop();

seed = t.elapsed_time().count();

unsigned int randomNumber(){
    const unsigned int a = 584958; //these were generated by keyboard mashing
    const unsigned int c = 43257802;
    const unsigned int m = 347582048;

    seed = (a * seed + c) % m;
    return seed;
}

void checkForMinigame(){
    randomNumber();
    checkMinigame = randomNumber() % 2 + 1;
}

Minigame:

For the minigame, I first tried making a branch class (which drew a rectangle and increased the y position) and including it in the player class, moving it every time the player moved – that didn’t work, it disappeared when the player wasn’t moving. Then it only moved when the player wasn't moving. Then I made a minigame engine class, which took a new player function (left and right, no animation, keeps things small) and the branch class to randomise where the branches would start: for each of the branch objects the minigame ran through and then checked collisions. To check for a collision, a loop ran through an array of the player x coordinates and the branch x coordinates, and if there was any overlap it took a life (seen in code snippet later in this entry). When lives were down to 0, the death scene was called and the game ended. This way means that lives reset between minigames, which probably shouldn’t. I’ll fix that by taking the lives variable outside the class, perhaps making a game engine class for the whole game to run through, although then I’d run into an error in calling each file in each other.

Fixing collisions with bounding boxes

29 April 2025

Program architecture:

Collision avoidance with the player and the wall has gone through several iterations:

The first of these was determining the state of the pixel in front by reading from the map array each time, which was very computationally inefficient.

The second involved calculating boundaries (as seen in previous entry) and using the centre point of the player to determine collision.

The third is an extension of the second, but it involves bounding boxes. The player is given a top, bottom, left and right coordinate, and these are checked against the boundaries of the map. This works well while going up and down the trunk, but fails to work properly in side passages, meaning I have a logic error somewhere. I used the centre point for the side passages as a work around; it is still not perfect, but at least it doesn’t push you outside the world any more.

The code below shows the creation and implementation of the bounding boxes: 


void Player::makeBoundingBox(int x, int y, char direction){

    x_pivot = x_pos + PlayerCentre;
    y_pivot = y_pos + PlayerCentre;

    if (direction == N || direction == S){
        top = y;
        bottom = y + 9;
        left = x + 2;
        right = x + 7;
    }
    else {
        top = y + 2;
        bottom = y + 7;
        left = x;
        right = x + 9;
    }
}

void Player::moveSprite(Joystick &joystick, N5110 &lcd, Map map){

    char direction = joystick.get_direction();
    makeBoundingBox(x_pos, y_pos, direction);

    if (((map.mapTile == 1 && 
     ((left >= 35 && right <= 46) || 
      (right < 46 && y_pivot >= 8 && y_pivot <= 16) || 
      (left > 35 && y_pivot >= 19 && y_pivot <= 28))) ||

    (map.mapTile == 2 && 
     ((left >= 35 && right <= 46) || 
      (left >= 35 && y_pivot >= 36 && y_pivot <= 44))) ||

    (map.mapTile == 3 && 
     ((left >= 35 && right <= 46) || 
      (right < 46 && y_pivot >= 23 && y_pivot <= 31) || 
      (left > 35 && y_pivot >= 11 && y_pivot <= 20))))
      ){

        moveForward(direction, lcd, map);
    } else {
        collisionDetected(direction, map, lcd);
    }
}

Project Highlights

30 April 2025

I keep walking in on my housemate playing final fantasy in the living room and I think it rubbed off a bit on this game, mostly in the cutscene, exploring and minigame elements. It was fun to research and talk to friends about game design and I got some ideas for mine from that, given I don't really play video games and wanted to design one from scratch.

Most impressive technical feature:

I spent a lot of time working on collision avoidance with the map. I’m really proud of how that ended up, with basic bounding boxes and a pivot point in the centre of the player, using an if else statement to move the player forwards or jump backwards in certain areas of the screen. It isn’t perfect, and occasionally the player can still glitch into the walls, but iterating through the different methods and getting more and more efficient was rewarding and fun. I learned a lot of what I’ve used here from designing the minigame, which needed to calculate overlap and worked in a bit of a different way – creating arrays from the x positions of the player and branch and determining overlap with for loops at a specific y position of the branch. Figuring out how to break out of several layers of for loops using a variable was a fun challenge too.

Most creative feature:

I used the introduction cutscene as part of my randomness generator; the time taken for the player to click through the text is the seed for the linear congruential method formula. This method isn’t great on actual randomness, but it’s efficient and I don’t need that much randomness for this game. Because the scene is called every time the game starts up, and has several different button presses, it’s different every time and difficult to game the system. I got the idea from researching how they did it in old arcade games. This way seems better than a list of numbers that just cycle through for the illusion of randomness. I also loved having creative freedom to work on the visual and story elements, discovered I really enjoy pixel art and want to make more games in future.

Reflection

30 April 2025

I have learned an enormous amount about C++ and object orientated programming in this project. About half way through, classes really clicked for me, and I now understand how to use them and why a lot better. Having a fun, creative project really helped me learn this – it meant I spent a lot of my free time thinking about it, more than I would have if the project was less personal.

In terms of Embedded Systems, I spent a lot of time trying to make my code more efficient, because I ran into some problems with space on the board, especially when trying to read large arrays. I didn’t use any other buttons, etc. because they weren’t needed and felt like they would have confused things. At the beginning I ran into some issues with the joystick button, and even now sometimes the game runs slowly or the joystick is read wrong, especially after I've been working on it for a long time. I also used the lcd for some testing – printing numbers to the screen when the serial monitor wasn’t working.

One of the main problems I had to solve was collisions, both with the minigame and the maze. I used the smaller scale of the minigame to help understand and figure out how I would do it for the rest of the game, and then adapted from that. The animations of the player meant that it moves slower than I’d like, and also two pixels at a time, which means it used to jump over the wall - more when it moved faster. This could be fixed by making the game tiled, but I played around with the logic there and it works pretty well now.

I pretty much started off by treating this as an art game, and I’m really proud of the animated elements, which made things like collision avoidance a lot harder. I did have more animations for the cutscenes, but unfortunately I accidentally deleted the originals. Lesson learned there. It was fun to design a totally new game, taking elements and ideas from existing ones, and playing with ways that the story could fit with the game mechanics.

I’m happy with how this project turned out. I enjoyed working on it, both the creative and logical elements, and it’s making me want to do more game design in future.

Final touches – fixing collisions

1 May 2025

Fixed collisions completely by updating response for the x and y direction (x direction puts the sprite in the centre, y direction is more complex because it has more options, and increased size of the falling branches to make the game more difficult.

Code snippet below deals with up-down collisions if the player is facing up. The same thing happens for facing down, and side to side is much simpler. Another way I could have done this was by taking the previous position and placing the sprite back there, but this is simple and doesn’t allow any glitching through walls. 


if (direction == N) {
    if (map.mapTile == 1 && right < 46){
        y_pos = 8;
    }
    else if (map.mapTile == 1 && left > 36) {
        y_pos = 20;
    }
    else if (map.mapTile == 2) {
        y_pos = 36;
    }
    else if (map.mapTile == 3 && right < 46) {
        y_pos = 24;
    }
    else if (map.mapTile == 3 && left > 36) {
        y_pos = 10;
    }
}

I’m really happy with this as a solution, rather than making the player move backwards, which often caused glitching through walls.
